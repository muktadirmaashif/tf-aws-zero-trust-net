**Core Architecture:**

### Phase 1: Foundation & State Management

#### Branch: `feature/remote-state-backend`

**Goal:** Establish a secure, locked remote state backend to enable team collaboration and prevent state corruption.

**Sub-steps:**
1.  Create a separate Terraform configuration for the backend infrastructure (S3 + DynamoDB).
2.  Apply it manually once to bootstrap the environment.
3.  Configure the root module to use this backend.

**File: `backend/main.tf`**
*   **What:** Defines an S3 bucket for state storage and a DynamoDB table for locking.
*   **Why:** Local state is insecure and doesn't support concurrency. Locking prevents two engineers from applying changes simultaneously, which causes state corruption.
*   **How:** Use `aws_s3_bucket` with `versioning` enabled and `aws_dynamodb_table` with a primary key `LockID`. Enable server-side encryption on the bucket.
*   **Industry Practice:** Never store state in public buckets. Enable MFA Delete on the state bucket for extra protection.
*   **Warning:** Do not commit the `backend.tf` file with hardcoded credentials. Use environment variables or AWS profiles.

**File: `root/backend.tf`**
*   **What:** The `terraform` block with `backend "s3"`.
*   **Why:** Tells Terraform where to read/write the state file.
*   **How:** Reference the bucket name and DynamoDB table created in the previous step. Set `encrypt = true`.
*   **Snippet:**
    ```hcl
    terraform {
      backend "s3" {
        bucket         = "my-org-tf-state-prod"
        key            = "networking/vpc-s3/terraform.tfstate"
        region         = "us-east-1"
        dynamodb_table = "my-org-tf-lock-table"
        encrypt        = true
      }
    }
    ```

### Phase 2: Secure VPC Module Development

#### Branch: `feature/vpc-module-core`

**Goal:** Create a reusable VPC module with private/public subnets and strict routing.

**Sub-steps:**
1.  Define module structure (`main`, `variables`, `outputs`, `versions`).
2.  Implement VPC, IGW, Subnets, and Route Tables.
3.  Implement `for_each` for subnet creation.

**File: `modules/vpc/variables.tf`**
*   **What:** Inputs for CIDR, AZs, Environment, and Tags.
*   **Why:** Standardizes the API of the module. Validation ensures bad data fails early.
*   **How:** Use `type = string` with `validation` blocks for CIDR formats. Use `map(string)` for tags.
*   **Industry Practice:** Always pin provider versions in `versions.tf`.

**File: `modules/vpc/main.tf`**
*   **What:** Resources for VPC, Subnets, Route Tables, and Associations.
*   **Why:** Separates public internet traffic from private internal traffic.
*   **How:**
    *   Create `aws_vpc` with `enable_dns_support/hostnames = true`.
    *   Create `aws_subnet` using `for_each` on a map of `{ name = cidr }`.
    *   Create `aws_route_table` for public (IGW route) and private (local only).
    *   Use `aws_route_table_association` to link subnets to tables.
*   **Warning:** Ensure private subnets do **not** have a route to the Internet Gateway.

**File: `modules/vpc/outputs.tf`**
*   **What:** Expose `vpc_id`, `private_subnet_ids`, `public_subnet_ids`.
*   **Why:** Other modules (like EKS or EC2) need these IDs to place resources.
*   **How:** Use simple output blocks. Mark sensitive outputs if necessary (though IDs are usually public).

#### Branch: `feature/vpc-security-hardening`

**Goal:** Add VPC Flow Logs and restrict Security Group egress.

**Sub-steps:**
1.  Create CloudWatch Log Group for Flow Logs.
2.  Create IAM Role for Flow Logs.
3.  Attach Flow Logs to VPC.
4.  Restrict Default Security Group.

**File: `modules/vpc/main.tf` (Additions)**
*   **What:** `aws_flow_log`, `aws_cloudwatch_log_group`, `aws_iam_role`.
*   **Why:** Flow logs are critical for security auditing and detecting anomalous traffic patterns (e.g., data exfiltration).
*   **How:**
    *   Create a CW Log Group with retention (e.g., 30 days).
    *   Create an IAM Role with `sts:AssumeRole` for `vpc-flow-logs.amazonaws.com`.
    *   Attach policy allowing `logs:PutLogEvents`.
    *   Create `aws_flow_log` referencing the VPC ID and Log Group ARN.
*   **Industry Practice:** Set retention policies on log groups to manage costs.

**File: `modules/vpc/main.tf` (Security Group)**
*   **What:** `aws_security_group` with restricted egress.
*   **Why:** Default SG allows all outbound traffic. Restricting it limits the blast radius if a resource is compromised.
*   **How:** Define explicit `egress` rules (e.g., only HTTPS 443). Remove the default "allow all" rule.
*   **Warning:** Be careful not to block essential AWS service endpoints if you aren't using VPC Endpoints yet.

#### Branch: `feature/vpc-private-link`

**Goal:** Enable private connectivity to S3 and STS using VPC Endpoints.

**Sub-steps:**
1.  Create Gateway Endpoint for S3.
2.  Create Interface Endpoint for STS.
3.  Associate Endpoints with Route Tables/Subnets.

**File: `modules/vpc/main.tf` (Additions)**
*   **What:** `aws_vpc_endpoint` resources.
*   **Why:** Keeps traffic between VPC and AWS services within the AWS network backbone. Improves security and performance. Eliminates need for NAT Gateway for S3/STS traffic.
*   **How:**
    *   **S3 Endpoint:** Type `Gateway`. Associate with private route table IDs.
    *   **STS Endpoint:** Type `Interface`. Associate with private subnet IDs and a specific Security Group. Enable `private_dns_enabled`.
*   **Industry Practice:** Use Interface Endpoints for services that don't support Gateway Endpoints (like STS, CloudWatch Logs, Secrets Manager).

**File: `modules/vpc/outputs.tf` (Additions)**
*   **What:** Output `s3_endpoint_id` and `sts_endpoint_id`.
*   **Why:** The S3 module needs the S3 Endpoint ID to restrict bucket policies.

### Phase 3: Secure S3 Module Development

#### Branch: `feature/s3-module-core`

**Goal:** Create a reusable S3 module with versioning, encryption, and lifecycle rules.

**Sub-steps:**
1.  Define complex variable structures for bucket configuration.
2.  Implement `for_each` for multiple buckets.
3.  Add versioning and encryption.

**File: `modules/s3/variables.tf`**
*   **What:** `bucket_config` as `map(object({...}))`.
*   **Why:** Allows defining multiple buckets with different settings in a single variable block. Highly scalable.
*   **How:** Define object attributes: `versioning_enabled`, `lifecycle_rules`, `tags`.

**File: `modules/s3/main.tf`**
*   **What:** `aws_s3_bucket`, `aws_s3_bucket_versioning`, `aws_s3_bucket_server_side_encryption_configuration`.
*   **Why:** Ensures data durability (versioning) and confidentiality (encryption).
*   **How:**
    *   Use `for_each` on `var.bucket_config` to create buckets.
    *   Use `for_each` on a filtered map for versioning (only if enabled).
    *   Enable SSE-KMS or AES256 encryption.
*   **Industry Practice:** Always enable versioning for production data buckets. Use KMS keys for higher security compliance.

**File: `modules/s3/main.tf` (Lifecycle)**
*   **What:** `aws_s3_bucket_lifecycle_configuration` with `dynamic` blocks.
*   **Why:** Manages data cost and compliance by expiring old objects or non-current versions.
*   **How:** Use `dynamic "rule"` inside the resource to iterate over `lifecycle_rules` from the variable.
*   **Warning:** Test lifecycle rules carefully. Accidental deletion can be catastrophic.

#### Branch: `feature/s3-security-hardening`

**Goal:** Enforce VPC-only access, block public access, and enable logging.

**Sub-steps:**
1.  Block all public access.
2.  Enforce HTTPS-only access.
3.  Restrict access to specific VPC Endpoint.
4.  Enable access logging.

**File: `modules/s3/main.tf` (Additions)**
*   **What:** `aws_s3_bucket_public_access_block`, `aws_s3_bucket_policy`.
*   **Why:** Prevents accidental data leaks. Ensures data is encrypted in transit. Restricts access to authorized networks only.
*   **How:**
    *   **Public Access Block:** Set all 4 flags to `true`.
    *   **HTTPS Policy:** Create a bucket policy with a `Deny` statement if `aws:SecureTransport` is `false`.
    *   **VPC Endpoint Policy:** Create a bucket policy with an `Allow` statement conditioned on `aws:sourceVpce` matching the VPC Endpoint ID passed from the root.
*   **Industry Practice:** The "Deny Non-SSL" policy is a CIS Benchmark requirement.

**File: `modules/s3/main.tf` (Logging)**
*   **What:** `aws_s3_bucket_logging`.
*   **Why:** Provides an audit trail of who accessed what data and when.
*   **How:** Target a separate logging bucket (passed via variable). Set prefix to `logs/<bucket-name>/`.
*   **Warning:** The target logging bucket must have a policy allowing the source bucket to write logs.

### Phase 4: Composition & Root Module

#### Branch: `feature/root-composition`

**Goal:** Wire VPC and S3 modules together in the root module.

**Sub-steps:**
1.  Define root variables.
2.  Instantiate VPC module.
3.  Instantiate Logging S3 bucket.
4.  Instantiate Data S3 buckets with VPC constraints.

**File: `root/main.tf`**
*   **What:** Module blocks for `vpc`, `log_bucket`, `data_buckets`.
*   **Why:** Orchestrates the infrastructure. Passes outputs from one module as inputs to another.
*   **How:**
    *   Call `module.vpc`.
    *   Call `module.log_bucket` (simple config).
    *   Call `module.data_buckets` passing `vpc_id` and `s3_endpoint_id` from `module.vpc`.
*   **Industry Practice:** Use `terraform.tfvars` to store environment-specific values (CIDRs, bucket names). Never hardcode secrets in `.tf` files.

**File: `root/outputs.tf`**
*   **What:** Output critical IDs for verification.
*   **Why:** Allows easy checking of created resources.
*   **How:** Output `vpc_id`, `data_bucket_arns`.

### Phase 5: DevSecOps Integration (Trivy)

#### Branch: `feature/devsecops-trivy`

**Goal:** Integrate static analysis security testing (SAST) for IaC using Trivy.

**Sub-steps:**
1.  Install Trivy.
2.  Create a scan script.
3.  Fix identified issues.
4.  Document the security posture.

**File: `scripts/scan.sh`**
*   **What:** A bash script to run Trivy against the Terraform code.
*   **Why:** Automates security checks. Catches misconfigurations (e.g., unencrypted buckets, open security groups) before deployment.
*   **How:**
    ```bash
    #!/bin/bash
    echo "Running Trivy IaC Scan..."
    trivy config --severity HIGH,CRITICAL .
    if [ $? -ne 0 ]; then
      echo "Security scan failed. Please fix issues."
      exit 1
    fi
    echo "Security scan passed."
    ```
*   **Industry Practice:** Run this script in CI/CD pipelines (GitHub Actions, GitLab CI) on every Pull Request.

**File: `README.md` (Update)**
*   **What:** Documentation on how to run security scans.
*   **Why:** Demonstrates commitment to security and provides instructions for other developers.
*   **How:** Add a section "Security Scanning" with instructions to install Trivy and run `./scripts/scan.sh`.
